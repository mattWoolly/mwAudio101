# GUI Architecture

This document is the single source of truth for the mwAudio101 editor: its JUCE
Component tree, the modern signal-flow vector look (maximum trademark distance, no
faceplate replica), resizable + hi-DPI rendering, the APVTS attachment binding
pattern, the preset browser view, the audio-decoupled telemetry path, and the
vector asset strategy. It realizes ADR-015 [ADR-015] verbatim and is bound by the
real-time invariants re-affirmed there and in ADR-008 [ADR-008].

Backlog tasks cite this document by section number (e.g. "implement per
docs/design/10-ui.md §4.2"). Parameter IDs, ranges, units, skews, and the
automatable/structural classification are **owned by the schema doc** (the
realization of ADR-008); this document **references** them and never re-mints them.
Where this document names an ID it is illustrative of a binding target only — the
authoritative list lives in the schema doc and the `ParamDefs` registry it
generates [ADR-008 §1, C1].

## 1. Scope and non-goals

### 1.1 In scope

- The single, format-agnostic `juce::AudioProcessorEditor` and its Component tree
  [ADR-015 Decision; ADR-011].
- The one logical design coordinate space and `AffineTransform` scaling for
  resize + hi-DPI [ADR-015 C1, C2].
- The signal-flow module decomposition (MODULATOR -> VCO -> SOURCE MIXER -> VCF ->
  VCA, plus CONTROLLER strip and transport/mode bar) and drawn patch lines
  [ADR-015 Decision].
- The custom `LookAndFeel` and single design-token table [ADR-015 C10].
- APVTS attachment binding for every control [ADR-015 C3, C4].
- The thin `PresetBrowser` editor view over the processor-owned `PresetManager`
  [ADR-015 C6].
- The one-directional, RT-safe UI<->engine boundary: APVTS write path and the
  lock-free SPSC telemetry read path [ADR-015 C5, C12].
- Repaint discipline (cached static layer, per-control dirty-rect, capped Timer)
  [ADR-015 C7].
- The "reduce motion / low-CPU" toggle and the OpenGL opt-in escape hatch
  [ADR-015 C8, C9].
- The vector-first asset strategy [ADR-015 C11].

### 1.2 Non-goals (owned elsewhere — referenced, not redefined)

- **Parameter IDs / ranges / skews / units / automatable classification** — owned
  by the schema doc / `ParamDefs` registry [ADR-008 §1-§4, C1-C7]. This document
  binds to them but does not define them.
- **Preset on-disk format, migration chain, factory bank content** — owned by the
  schema doc [ADR-008 §5-§7]. The `PresetManager` *contract this document depends
  on* is summarized in §9 but its serialization internals are not redefined here.
- **State/preset load-failure runtime behavior** — owned by ADR-021. This document
  defines only the non-modal warning *surface* (§9.4) that ADR-021's failure path
  drives.
- **DSP, voice rendering, FX, arp/seq engine semantics** — owned by their design
  docs. The editor never calls DSP directly (§8).
- **Naming / legal disclaimer text** — owned by the naming/legal track
  [research/12 §5.5, §7.1; ADR-015 Owner ratification]. The editor surfaces a
  disclaimer string but does not author the legal posture.

## 2. Design principles (normative)

These are the invariants every section below upholds. They restate ADR-015 so a
fresh implementer need not re-read the ADR to know the hard rules.

1. **One coordinate space, no pixel coordinates.** All layout math is expressed in
   logical design units over a fixed-aspect canvas; physical pixels are reached
   only via a single `AffineTransform`. Hard-coded pixel coordinates are prohibited
   [ADR-015 C1; Decision].
2. **Vector-only, no raster faceplate.** Every control is drawn in code; the only
   bundled art is SVG (logo + static decoration). No `@2x/@3x` raster matrix exists
   [ADR-015 C11].
3. **Maximum trademark distance by construction.** The look is a signal-flow
   diagram in a non-Roland palette, not a recolored SH-101 faceplate
   [research/12 §5.2, §5.4, §7.2; ADR-015 Decision]. The palette is **never** Roland
   grey/red/blue [ADR-015 C10].
4. **One design-token table.** Palette, stroke weights, corner radii, and
   typography live in exactly one table; swapping it reskins the whole plugin
   without touching layout or binding code [ADR-015 C10].
5. **APVTS-only write path.** The editor mutates engine state only through APVTS
   attachments. It holds zero audio-domain state and never calls processor DSP
   [ADR-015 C3, C4; ADR-008 C19].
6. **Lock-free SPSC-only read path.** Audio-derived visuals come only from a
   pre-allocated, fixed-capacity, lock-free single-producer/single-consumer
   snapshot the audio thread writes to without blocking or allocating
   [ADR-015 C5, C12].
7. **No whole-editor repaint.** Static layers are cached; only moving controls
   invalidate, each within its own bounds [ADR-015 C7].
8. **Software render path first; OpenGL opt-in, OFF by default** so the Linux x64
   hard gate never depends on a GPU driver [ADR-015 C9].
9. **GUI is not pixel-identical across platforms** — only the audio is bit-exact
   [ADR-015 Decision/Consequences; ADR-011]. UI tests assert *behavior and layout
   geometry*, never pixel equality.

## 3. Real-time invariants (audio-thread contract)

The editor runs entirely on the JUCE **message thread**; it never executes on the
audio thread. The following are binding and testable [ADR-015 C12; ADR-008 C19].

1. The audio callback holds **no reference** to any GUI object.
2. The audio callback takes **no lock** that the message thread can hold, and the
   editor takes no lock the audio thread holds.
3. The audio callback performs **no heap allocation** on behalf of the GUI.
4. The only audio->GUI channel is the pre-allocated SPSC telemetry snapshot (§8.3):
   the audio thread is the single producer and only writes; it never blocks or
   allocates; an overrun is dropped, not waited on.
5. A GUI frame overrun (missed Timer deadline, slow repaint) degrades **visuals
   only** and can **never** cause an xrun.
6. The editor allocates its Component tree, attachments, cached layers, and SPSC
   consumer buffers in its constructor / `resized()` on the message thread — never
   from a render or Timer callback hot path beyond bounded, amortized work.

Hot-path methods on the telemetry producer side and any method the audio thread may
touch are declared `noexcept` (§8.3).

## 4. Coordinate space, scaling, and resize

### 4.1 Logical design space

The editor is laid out over one logical design coordinate space. Nominal extent is
**1000 x 640 design units** (`kDesignWidth = 1000.0f`, `kDesignHeight = 640.0f`);
the final aspect ratio is fixed during the visual-design pass and then frozen
[ADR-015 Decision]. All sub-component bounds are computed from this space in
`resized()`. There are **zero hard-coded pixel coordinates** [ADR-015 C1].

The aspect ratio constant is centralized:

```cpp
namespace mw::ui {
inline constexpr float kDesignWidth  = 1000.0f;   // design units
inline constexpr float kDesignHeight = 640.0f;    // design units
inline constexpr float kAspectRatio  = kDesignWidth / kDesignHeight; // (PI)
}
```

The exact `kDesignWidth`/`kDesignHeight`/`kAspectRatio` values are a **(PI)**
pragmatic invention pending the visual-design pass; they centralize in the
calibration table (`core/calibration/Calibration.h`) so the design pass can retune
the canvas without touching layout code.

### 4.2 AffineTransform scaling (the canonical hi-DPI/resize mechanism)

Scaling from design units to physical pixels uses **one** `AffineTransform`,
recomputed only in `resized()`. JUCE's deprecated `setScaleFactor` /
per-pixel-density path is **explicitly rejected** [ADR-015 Decision].

```cpp
// In MwAudioEditor::resized():
const auto bounds = getLocalBounds().toFloat();
const float scale = juce::jmin(bounds.getWidth()  / kDesignWidth,
                               bounds.getHeight() / kDesignHeight);
designToPixels = juce::AffineTransform::scale(scale)
                     .translated(/* centering offsets in pixels */);
```

Each sub-component receives its bounds in design units; the root applies
`designToPixels`. Because vector paths re-rasterize at the physical-pixel
transform, hi-DPI is automatic and consistent across JUCE's macOS / Linux / Windows
backends with no raster asset swap [ADR-015 C1, C11].

`getDesignToPixels()` and the current `getScaleFactor()` are exposed so tests can
assert geometry (§13).

### 4.3 Resizable window + constrainer

The editor enables resize with a constrainer enforcing the fixed aspect ratio
[ADR-015 C2, Decision].

```cpp
// MwAudioEditor constructor:
setResizable (/*useBottomRightCornerResizer*/ true, /*useConstrainer*/ true);
constrainer.setFixedAspectRatio (kAspectRatio);                 // ADR-015 C1
constrainer.setSizeLimits (minWpx, minHpx, maxWpx, maxHpx);
setConstrainer (&constrainer);
```

| Property | Value | Unit | Notes | Source |
| --- | --- | --- | --- | --- |
| Aspect ratio | fixed = `kAspectRatio` | ratio | enforced by constrainer | [ADR-015 C1] |
| Min logical scale | 0.75 (75%) | ratio | smallest snap preset | [ADR-015 C2] |
| Max logical scale | 2.00 (200%) | ratio | largest snap preset | [ADR-015 C2] |
| Default logical scale | 1.00 (100%) | ratio | initial size = design extent | (PI) |
| Snap presets | 75 / 100 / 150 / 200 | % | scale-menu options | [ADR-015 C2] |

The default scale, min/max pixel limits, and the snap-preset list are **(PI)** and
centralize in `core/calibration/Calibration.h`.

### 4.4 Scale presets and persisted size

A scale-preset selector (in the transport/mode bar, §6.6) snaps the window to a
logical scale (75/100/150/200%) by setting bounds to `scale * kDesignWidth` x
`scale * kDesignHeight` in physical pixels at the current display density
[ADR-015 C2].

The last window size is persisted in **APVTS state** so it round-trips on session
reload [ADR-015 C2, Decision]. Per the schema contract, UI size lives in the
`<extras>` subtree of the canonical ValueTree, **not** as a host parameter
[ADR-008 §4, §5, C8]. The editor:

- On construction, reads the stored size (if present) from the `<extras>` UI node
  via the processor's accessor and calls `setSize(...)`; absent => default scale.
- On `resized()` (debounced), writes the new size back to the `<extras>` UI node on
  the message thread.

The exact `<extras>` node/key names are defined by the schema doc; this document
binds to them through a narrow processor accessor pair
(`getStoredEditorSize()` / `setStoredEditorSize(juce::Point<int>)`) so the editor
never reaches into the ValueTree layout directly.

### 5.3 Module base and the signal-flow decomposition

The panel mirrors the documented signal chain so the UI teaches the architecture
[ADR-015 Decision; research/12 §7.3 transparency lever]. Each module is a
`juce::Component` that owns its controls, their APVTS attachments, and a
`layoutDesignUnits()` helper invoked from the parent `resized()`.

```cpp
// ui/modules/ModuleBase.h
class ModuleBase : public juce::Component
{
public:
    ModuleBase (juce::AudioProcessorValueTreeState& state, juce::StringRef title)
        : apvts (state), moduleTitle (title) {}

    // Lay out children in design units; parent passes a design-space rectangle.
    virtual void layoutDesignUnits (juce::Rectangle<float> designBounds) = 0;

protected:
    juce::AudioProcessorValueTreeState& apvts;
    juce::String moduleTitle;
};
```

Module responsibilities (controls listed are illustrative binding targets; the
authoritative IDs/ranges are owned by the schema doc [ADR-008 §1-§4]):

| Module | Signal-chain role | Representative bound parameters (IDs owned by schema doc) |
| --- | --- | --- |
| `ModulatorModule` | MODULATOR (LFO / S&H + depth) | `mw101.lfo.rate`, `mw101.lfo.shape`, mod-depth controls |
| `VcoModule` | VCO + sub + noise source | `mw101.vco.range`, waveform mix, `mw101.vco.pwm`, `mw101.sub.mode`, pitch |
| `SourceMixerModule` | SOURCE MIXER | saw / pulse / `mw101.sub.level` / noise levels |
| `VcfModule` | VCF | `mw101.vcf.cutoff`, `mw101.vcf.resonance`, env amount, kbd track, mod |
| `VcaModule` | VCA + envelope | env gate/select, level, env A/D/S/R |
| `ControllerStrip` | CONTROLLER | `mw101.glide.time`, bend sensitivity, mod routing, transpose |
| `TransportModeBar` | transport / arp / seq mode | arp mode, tempo-sync subdivisions, run/hold |

Modules are connected by drawn vector **patch lines** rendered into the cached
background layer (§7.2). Software-only extensions (Sine LFO shape, 32'/64' VCO
registers) appear only as appended choice entries the schema fences as
`sound_ext`; the UI must visually mark them as extensions and **never** present
them as 1982 hardware behavior [ADR-008 §7, C6, C15; research/12 §3.1].

Thin `juce::Slider` / `juce::Button` / `juce::ComboBox` subclasses carry their own

is not protectable; distinctiveness lives in styling) [research/12 §7.2;

`ChoiceSelector` (a `ComboBox` subclass) renders software-only entries with the
`extensionTag` token so `sound_ext` options are visually fenced from hardware
options [ADR-008 §7, C6, C15].

## 7. Repaint discipline

The real cost is on the message thread, not the audio thread [ADR-015 C7,
Repaint discipline]. The regime below is binding; a stray full-editor `repaint()`
is a frame-cost regression to reject in review [ADR-015 Consequences].

### 7.1 Static-layer cache

Panel chrome, module outlines, patch lines, and static labels are rendered **once**
into a `juce::Image` (the `BackgroundLayer`) and regenerated **only on resize**
[ADR-015 C7].

```cpp
// ui/BackgroundLayer.h
class BackgroundLayer final : public juce::Component
{
public:
    void regenerate (juce::Rectangle<int> pixelBounds,
                     const juce::AffineTransform& designToPixels,
                     const DesignTokens&);   // called from MwAudioEditor::resized()
    void paint (juce::Graphics&) override;   // blits the cached image; no path work
private:
    juce::Image cached;                      // chrome + patch lines + labels
};
```

### 7.2 Patch lines

Patch lines connecting modules (MODULATOR -> VCO -> SOURCE MIXER -> VCF -> VCA) are
vector paths drawn into the cached `BackgroundLayer`, not re-stroked per frame
[ADR-015 Decision, C7]. They are static art (the routing topology is fixed), so
they never animate on the Timer.

### 7.3 Per-control dirty-rect repaint

Each control invalidates only its own bounds on a value change. Whole-editor
`repaint()` is **disallowed** [ADR-015 C7]. Parameter-change repaints are coalesced
(JUCE already coalesces invalidations within a message loop pass; controls must not
force synchronous repaints).

### 7.4 Telemetry repaint cap

Meter/scope animation repaints are driven exclusively by the coalescing Timer at
30-60 Hz (§8.4), never by per-sample or per-block callbacks [ADR-015 C5].

## 8. UI<->engine decoupling (one-directional, RT-safe)

The boundary is one-directional and RT-safe by construction [ADR-015
Decoupling, C3, C4, C5, C12; ADR-008 C19].

### 8.1 Write path — APVTS attachments only

Every control binds through a JUCE APVTS attachment. The editor holds **zero**
audio-domain state and never calls processor DSP [ADR-015 C3, C4].

```cpp
// Inside a module, per control:
std::unique_ptr<juce::AudioProcessorValueTreeState::SliderAttachment>   cutoffAttach;
std::unique_ptr<juce::AudioProcessorValueTreeState::ButtonAttachment>   gateAttach;
std::unique_ptr<juce::AudioProcessorValueTreeState::ComboBoxAttachment> lfoShapeAttach;

// Construction (paramID strings are owned by the schema registry, ADR-008 C1):
cutoffAttach = std::make_unique<juce::AudioProcessorValueTreeState::SliderAttachment>(
                   apvts, ParamIds::vcfCutoff, cutoffSlider);
```

Attachment binding rules:

- The `paramID` string passed to each attachment is referenced from the schema
  registry's symbol set (a generated `ParamIds` namespace / constants owned by the
  schema doc) — the editor **never** hard-codes a raw `"mw101.*"` literal it
  defines itself [ADR-008 C1].
- Attachments are created in module construction (message thread) and destroyed in
  module destruction; they are never created or destroyed from a render/Timer hot
  path (RT invariant §3.6).
- **Structural** parameters (`mw101.quality`, `mw101.voice.mode`,
  `mw101.voice.count`, `mw101.unison.count`) are `withAutomatable(false)` per the
  schema [ADR-008 §4, C7]. Their editor controls still bind via attachments, but
  the editor must not assume they are sweepable; they reconfigure the engine via
  the processor's prepare-style path. The editor only *requests* the value change;
  the reconfiguration is the processor's concern, off the audio thread.

### 8.2 Read path for parameter values — APVTS listeners, never engine polling

When the host automates or recalls a parameter, the APVTS attachment moves the
control to match. The editor reads parameter changes only via APVTS listeners /
attachments, **never** by polling the engine [ADR-015 C4]. This yields full
automation, host undo, and DAW state save/restore with no bespoke glue
[ADR-015 Decision; ADR-008 §4].

### 8.3 Read path for audio-derived visuals — lock-free SPSC snapshot

Meters, scope samples, and mod-source indicators flow one-way through a
**pre-allocated, fixed-capacity, lock-free single-producer/single-consumer**
snapshot. The audio thread is the single producer and only writes (no blocking, no
allocation); the editor's Timer is the single consumer [ADR-015 C5, C12;
ADR-008 C19].

```cpp
// ui/Telemetry.h
namespace mw::ui::Telemetry {

struct Snapshot                                  // POD, fixed size, trivially copyable
{
    float vcaLevelL = 0.0f;                      // post-VCA RMS/peak, 0..1
    float vcaLevelR = 0.0f;
    float vcfCutoffDisplay = 0.0f;               // current modulated cutoff, 0..1
    std::array<float, kScopePoints> scope {};    // decimated waveform for the scope
    std::uint32_t lfoPhase = 0;                  // for mod-source indicator
    std::uint64_t seqStep  = 0;                   // current sequencer step (display only)
};

inline constexpr int kScopePoints = 256;          // (PI)
inline constexpr int kFifoCapacity = 4;           // (PI) ring depth

// Single-producer end (audio thread). All methods noexcept, no alloc, no lock.
class Producer {
public:
    void prepare (/* sized in processor::prepareToPlay */) noexcept;
    void push (const Snapshot&) noexcept;        // overrun => overwrite oldest, never block
};

// Single-consumer end (message thread / editor Timer).
class Consumer {
public:
    bool pull (Snapshot& out) noexcept;          // coalesces to most-recent; false if none
};
} // namespace mw::ui::Telemetry
```

Sizing and ownership rules:

- The SPSC buffer is owned by the **processor** and sized in `prepareToPlay`
  (pre-allocated, fixed capacity). The editor receives a `Consumer` view; if the
  editor is closed there is simply no consumer — the producer keeps writing into
  the ring harmlessly (or is gated by an atomic "editor open" flag, an
  implementation choice that must not introduce a lock) [ADR-015 C5].
- `kScopePoints`, `kFifoCapacity`, and the per-frame decimation factor are **(PI)**
  and centralize in `core/calibration/Calibration.h`.
- An overrun (producer faster than consumer) drops/overwrites the oldest frame and
  **never** blocks the audio thread [ADR-015 C5; §3.4].

### 8.4 Coalescing telemetry Timer

A single `juce::Timer` on the editor drains the SPSC at **30-60 Hz**, takes the
most-recent `Snapshot`, and triggers **targeted** repaints on the `ScopeMeterOverlay`
and any affected mod-source indicators only [ADR-015 C5, C7].

| Property | Value | Unit | Notes | Source |
| --- | --- | --- | --- | --- |
| Timer rate | 30-60 | Hz | one coalescing Timer; default 60, floor 30 | [ADR-015 C5] |
| Default rate | 60 | Hz | (PI) default within the allowed band | (PI) |
| Reduce-motion rate | 0 (suppressed) or downsampled | Hz | see §10 | [ADR-015 C8] |

The default 60 Hz and the reduce-motion downsample factor are **(PI)** within the
ADR-mandated 30-60 Hz band and centralize in `core/calibration/Calibration.h`.

## 9. Preset browser

### 9.1 Ownership boundary

The `PresetManager` lives in the **processor**, not the editor [ADR-015 C6;
ADR-008 §6]. The editor's `PresetBrowser` is a **thin view** that lists, filters,
and loads by calling the manager and listening to a change broadcaster
[ADR-015 Decision].

```cpp
// ui/PresetBrowser.h
class PresetBrowser final : public juce::Component,
                            private juce::ChangeListener
{
public:
    explicit PresetBrowser (PresetManager& manager);   // manager owned by processor
    ~PresetBrowser() override;

    void layoutDesignUnits (juce::Rectangle<float> designBounds);
    void resized() override;

private:
    void changeListenerCallback (juce::ChangeBroadcaster*) override; // refresh on load/save
    void refreshList();                                              // re-read names/categories
    void loadSelected();                                             // -> manager.loadPreset(idx)

    PresetManager& manager;
    juce::ListBox    presetList;        // names + category from manager
    juce::ComboBox   categoryFilter;    // schema taxonomy (ADR-008 C14)
    juce::TextButton prevButton, nextButton, loadButton;
};
```

### 9.2 PresetManager contract this view depends on

The full `PresetManager` is specified by the schema doc / ADR-008 §6; the editor
depends only on this narrow surface (it does not redefine serialization, migration,
or the on-disk format):

```cpp
// processor-owned; declared by the schema/state doc, summarized here for the view:
class PresetManager {
public:
    int                getNumPresets() const;
    juce::String       getPresetName (int index) const;
    juce::String       getPresetCategory (int index) const;   // ADR-008 C14 taxonomy
    int                getCurrentPresetIndex() const;
    void               loadPreset (int index);                // message thread only
    void               loadNext();  void loadPrev();
    juce::ChangeBroadcaster& getBroadcaster();                // notifies the view
};
```

### 9.3 Load is message-thread only; no tree pointer crosses to audio

`PresetManager::loadPreset` builds/swaps the state tree on the **message thread
only** and hands the audio thread parameter **values** via APVTS atomics — never a
tree pointer — so a preset load never allocates or frees on the audio thread
[ADR-015 C6; ADR-008 §5, §6, C19]. The non-APVTS `<extras>` (100-step pattern, arp
latch) is handed over via the schema's pre-allocated lock-free SPSC double-buffer
swap [ADR-008 C19, C20]. The editor's role is purely to invoke `loadPreset(index)`
on the message thread and refresh its list on the broadcaster callback.

The category filter combo enumerates the schema taxonomy: `AcidBassLead`,
`SubBass`, `Lead`, `PWMStrings`, `BlipsFX`, `SeqArpRiff` [ADR-008 C14]. These
strings are referenced from the schema doc's category enum, not re-minted here.

### 9.4 Load-failure surface (driven by ADR-021)

When a state/preset load fails, ADR-021 mandates a **non-modal, message-thread**
warning — never a modal dialog during host project load. This document owns only
the *surface*: a `StatusBanner` component (§5.1) that the processor's load-failure
path drives via a thread-safe `AsyncUpdater`/`ChangeBroadcaster` notification. The
banner text, severity, and dismissal are message-thread only and never block
[ADR-021 Context/forces]. The banner also hosts the static non-affiliation
disclaimer string (authored by the legal track, not here) [research/12 §5.5, §7.1].

## 10. Reduce-motion / low-CPU toggle

A "reduce motion / low-CPU" toggle (in the transport/mode bar) suppresses or
downsamples meter/scope animation for the worst case (weak GPUs, very high-res
displays) [ADR-015 C8]. Control bindings and automation remain **fully functional**
when the toggle is on [ADR-015 C8].

Behavior:

- ON => the telemetry Timer either stops or drops to a low rate, and the
  `ScopeMeterOverlay` paints a static/idle state. No control attachment is affected.
- The toggle state persists in the `<extras>` UI subtree (a UI preference, not a
  host parameter) [ADR-008 §4, §5, C8].

## 11. Rendering backend: software-first, OpenGL opt-in

The software render path is primary and must meet the frame budget without a GPU
[ADR-015 C9]. A `juce::OpenGLContext` is an **opt-in escape hatch that is OFF by
default** [ADR-015 C9].

- `juce::OpenGLContext openGLContext;` is a member that is **not** attached by
  default. It is attached only when an explicit user/advanced setting requests it.
- The **Linux x64 hard gate must not require OpenGL** — CI for the hard gate runs
  the software path [ADR-015 C9; ADR-011 platform tiers].
- GUI rendering is **not** guaranteed pixel-identical across platforms
  (software-path antialiasing/gradients vary); only the audio is bit-exact
  [ADR-015 Decision/Consequences; ADR-011]. Tests assert geometry/behavior, not
  pixels (§13).

## 12. Asset strategy (vector-first)

The only bundled art is **SVG** (logo + static decorative art), loaded via
`juce::Drawable::createFromSVG` and compiled through `BinaryData`
[ADR-015 C11].

- There are **zero** raster faceplate bitmaps and **no** `@2x/@3x` raster matrix in
  the build [ADR-015 C11].
- SVG files live under `ui/assets/*.svg` and are added to the JUCE `BinaryData`
  target by CMake (the build doc owns the embedding mechanics).
- Because controls and chrome are coded vector paths, hi-DPI is automatic; SVG is
  reserved for art that is genuinely static (logo, decorative glyphs)
  [ADR-015 C11; Decision].

## 13. Testing strategy notes

UI tests run headless on the message thread and assert **behavior and geometry**,
never pixel equality (the GUI is not pixel-identical across platforms — §11)
[ADR-015 Decision/Consequences].

- Geometry: construct the editor at several sizes, assert
  `getDesignToPixels()` maps known design points to expected pixel points within
  tolerance and that the aspect ratio is held by the constrainer (§4).
- Binding: drive a control, assert the bound APVTS parameter changed; change the
  APVTS parameter from the host side, assert the control moved (§8.1, §8.2).
- No-engine-call: a test double processor asserts the editor never calls a DSP
  method directly (only APVTS + telemetry) (§8).
- Telemetry RT-safety: an instrumented producer asserts `push()` performs no
  allocation and no lock; an overrun is dropped, not blocked (§8.3) [ADR-015 C12].
- Repaint hygiene: assert no code path invokes whole-editor `repaint()`; static
  layer regenerates only on resize (§7) — enforceable via a paint-count probe.
- Preset view: selecting a preset invokes `PresetManager::loadPreset` on the
  message thread; the list refreshes on the broadcaster callback (§9).

## Acceptance hooks

A backlog task's tests MUST verify the following objectively-testable properties:

- **Single coordinate space / no pixel coordinates.** The editor lays out entirely
  in design units; resizing changes only the single `AffineTransform`; no
  sub-component stores absolute pixel coordinates. (§4) [ADR-015 C1]
- **Resize + DPI crispness via AffineTransform.** Re-rendering at a new
  size/DPI uses one `AffineTransform` over the design space with no raster asset
  swap; the constrainer holds the fixed aspect ratio. (§4) [ADR-015 C1]
- **Scale presets persist.** Picking 75/100/150/200% snaps the window and the new
  size round-trips via the `<extras>` UI node on session reload. (§4.4)
  [ADR-015 C2; ADR-008 §5]
- **Control -> parameter via APVTS only.** Moving any control writes its bound
  parameter through an APVTS attachment; the editor makes no direct DSP call.
  (§8.1) [ADR-015 C3]
- **Host automation -> control via listener only.** Host-driven parameter changes
  move the control via the attachment; the editor never polls the engine. (§8.2)
  [ADR-015 C4]
- **Telemetry is lock-free SPSC, pre-allocated.** The audio-thread `push()`
  allocates nothing, takes no lock, and drops on overrun; the message-thread Timer
  drains at 30-60 Hz and triggers only targeted repaints. (§8.3, §8.4)
  [ADR-015 C5, C12]
- **Preset load is message-thread only.** Loading from the browser invokes the
  processor-owned `PresetManager` on the message thread; no tree pointer and no
  alloc/free crosses to the audio thread. (§9) [ADR-015 C6; ADR-008 C19]
- **No whole-editor repaint; cached static layer.** Only moving controls
  invalidate their own bounds; the background (chrome + patch lines + labels)
  regenerates only on resize. (§7) [ADR-015 C7]
- **Reduce-motion toggle.** Enabling it suppresses/downsamples meter/scope
  animation while leaving all control bindings and automation functional. (§10)
  [ADR-015 C8]
- **OpenGL OFF by default.** No `OpenGLContext` is attached by default; the
  software path meets budget; the Linux x64 hard gate does not require OpenGL.
  (§11) [ADR-015 C9]
- **Single design-token reskin.** Swapping the `DesignTokens` table restyles every
  control without touching layout or binding code; the shipped palette is never
  Roland grey/red/blue. (§6.1) [ADR-015 C10; research/12 §7.2]
- **Vector-only assets.** Only static SVG art and the logo exist; no raster
  faceplate bitmaps and no `@2x/@3x` raster matrix are present in the build.
  (§12) [ADR-015 C11]
- **Audio-thread holds no GUI state.** Under any UI activity the audio callback
  holds no GUI reference, takes no lock, and performs no GUI heap allocation; a GUI
  frame overrun degrades visuals only and never causes an xrun. (§3) [ADR-015 C12;
  ADR-008 C19]
- **Software-extension fencing.** Software-only options (Sine LFO, 32'/64'
  registers) are visually marked as extensions and never presented as 1982
  hardware behavior. (§5.3, §6.3) [ADR-008 §7, C6, C15]
